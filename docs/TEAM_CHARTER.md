# 团队组织手册 - Claude Code 电子宠物项目

## 1. 团队架构

### 角色定义
- **项目经理(我)**: 接收需求、组织团队、验收交付
- **架构师**: 技术方案设计、代码审查
- **开发工程师**: 编码实现、自测验证
- **测试工程师**: 测试用例编写、执行验证

### 协作流程
```
需求接收 → 技术设计 → 编码实现 → 自测验证 → 迭代修复 → 交付验收
```

## 2. 工作原则

### 核心原则
1. **TDD驱动**: 先写测试，再写代码
2. **交付导向**: 100%测试通过才能汇报
3. **自测验证**: 开发必须自测后再提交
4. **迭代修复**: 测试失败立即修复

### 禁止事项
- ❌ 不自测就提交
- ❌ 测试失败不修复就汇报
- ❌ 交付Web方案而不是桌面exe

## 3. 交付标准

### MVP交付要求
- ✅ 真正的Windows exe可执行文件
- ✅ 无需启动服务器，双击运行
- ✅ 宠物动画正常显示
- ✅ 状态切换功能正常
- ✅ 点击交互响应正常

### 验收条件
- 100%自动化测试通过
- 功能演示无bug
- 代码审查通过

## 4. 技术栈

### 桌面应用框架
- **Electron**: 跨平台桌面应用框架
- **Node.js**: JavaScript运行时
- **Native API**: Windows原生能力

### 项目结构
```
pets/
├── pet-desktop/           # 桌面应用主目录
│   ├── src/
│   │   ├── main.js      # Electron主进程
│   │   ├── preload.js   # 预加载脚本
│   │   └── renderer.js   # 渲染进程
│   ├── assets/
│   │   └── sprites/     # 动画帧
│   ├── package.json
│   └── build/           # 打包输出
└── docs/
    └── 测试报告/
```

## 5. 质量保证

### 测试要求
- 每个功能必须有测试用例
- 测试覆盖率 > 80%
- 所有测试通过后才能交付

### 代码规范
- ESLint代码检查
- 统一的代码风格
- 必要的注释
